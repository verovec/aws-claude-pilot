# Infrastructure Agent: AWS Claude Pilot

This file is the Terraform and deployment knowledge base. It is read by the `/sync` command and by any agent performing infrastructure work.

## Component Registry

Each component is a logical group of ECS services sharing the same YAML source directory, Terraform prefix, and locals block. `/sync` iterates over every component listed here, processing all environments.

### app (template example)

| Key | Value |
|-|-|
| YAML source | `deploy/<env>/` |
| Terraform prefix | `app` |
| Service name prefix | `<project>-app` (uses `var.project`) |
| Locals env | `local.app_env` |
| Locals secrets | `local.app_secrets` |
| Monitoring module | `module "monitoring"` |
| ALB listener priority | `100` |

**Name Mapping:**

| YAML service.name | Terraform module name | Variable prefix | YAML filename | Health check |
|-|-|-|-|-|
| myapp-app | app | app | app.yaml | /health |

For single-service components the module name equals the prefix, so variables are `app_image`, `app_cpu`, etc. For multi-service components, suffix is appended: `<prefix>_<suffix>_image`.

**Docker image tags:** Uses `app-latest`.

**Known Terraform-Expression Env Vars:**

| Env var | Terraform value | Notes |
|-|-|-|
| `APP_ENVIRONMENT` | `var.environment` | YAML has literal env name; TF uses variable |

To add a new component, create a new section here with its YAML source, prefix, locals, name mapping, and known TF-expression vars. Then create the locals block in `terraform/main.tf` and the YAML files in the deploy directory.

## Terraform Layout

All paths relative to `terraform/`.

### Module Inventory

| Module | Purpose |
|--------|---------|
| `modules/kms` | Encryption key for RDS, S3, CloudWatch |
| `modules/vpc` | VPC, subnets, NAT gateways, security groups |
| `modules/ecs` | ECS Fargate cluster + CloudWatch log group |
| `modules/iam` | ECS task roles, GitHub OIDC, ECR repos, CI/CD roles |
| `modules/rds` | PostgreSQL RDS instances with auto-generated credentials |
| `modules/s3` | Encrypted S3 buckets with public access block |
| `modules/acm` | ACM certificate with Route53 DNS validation (optional) |
| `modules/alb` | Application Load Balancer with HTTPS + HTTP redirect |
| `modules/ecs-service` | ECS Fargate service with optional ALB integration |
| `modules/app-secret` | Secrets Manager secret (data-source lookup or placeholder) |
| `modules/monitoring` | CloudWatch dashboard (ECS CPU/memory, RDS, logs) |
| `modules/bastion` | SSM-managed bastion for RDS access |

### Scope Boundary

`/sync` only operates within each component's ECS section of `main.tf`.

**In scope** (generated/updated by `/sync`):
- `module "<prefix>_secret"` -- app secret resource (creates with placeholders); `/sync` verifies block exists
- `module "<prefix>_<additional_secret>"` -- additional secret modules (auto-managed by `/sync`, see Additional Secret Module Detection)
- `locals` block -- `<prefix>_secrets` map reconciled by `/sync` from YAML secret mappings (union of all YAML files); `<prefix>_env` map reconciled by `/sync` from YAML `env` sections (union of all YAML files, known TF-expression vars preserved); ARN definitions manually maintained
- `module "<prefix>_*"` -- ECS service module blocks (generated)
- `module "<prefix>_monitoring"` or `module "monitoring"` -- `ecs_services` map (regenerated)
- `variables.tf` -- service image/cpu/memory/desired_count variables
- `environments/<env>.tfvars` -- service variable values

**Out of scope** (never touch):
- `module "kms"`, `module "vpc"`, `module "ecs"`, `module "iam"` -- shared platform
- `module "rds"`, `module "s3"` -- data stores
- `module "acm"`, `module "alb"` -- shared resources
- `module "bastion"` -- operational tooling
- `providers.tf`, `backend.tf`, `outputs.tf`, `locals.tf` -- root config
- `modules/` -- module source code

### main.tf Structure

```
module "kms"                              (OUT OF SCOPE)
module "vpc"                              (OUT OF SCOPE)
module "ecs"                              (OUT OF SCOPE)
module "iam"                              (OUT OF SCOPE)
module "rds"                              (OUT OF SCOPE)
module "s3"                               (OUT OF SCOPE)
module "acm"                              (OUT OF SCOPE)
module "alb"                              (OUT OF SCOPE)

# --- <component> section ---
module "<prefix>_secret"                  (IN SCOPE - secret resource with placeholders, /sync verifies block exists)
module "<prefix>_<additional_secret>"     (IN SCOPE - auto-managed by /sync)
locals { <prefix>_* }                     (IN SCOPE - secrets map + env map reconciled by /sync; ARN definitions manually maintained)
module "<prefix>_*"                       (IN SCOPE - generated by /sync)
module "<prefix>_monitoring"              (IN SCOPE - ecs_services map regenerated by /sync)

module "bastion"                          (OUT OF SCOPE)
```

### Additional Secret Module Detection

YAML secrets reference different kinds of sources via their group name:

| Kind | YAML group name | Terraform owner |
|-|-|-|
| RDS credentials | `rds` | `module.rds[...]` (out of scope) |
| App secret | `app` | `module.<prefix>_secret` (static) |
| Additional secrets | any other name | auto-managed by `/sync` |

When `/sync` reads all YAML files for a component, it collects every secret group name that is not `rds` and not `app`. Each distinct additional secret group maps to its own Terraform module.

**Deriving the module name from the YAML group:**

The YAML group name (e.g. `celery-certs`) is converted to the Terraform identifier by replacing hyphens with underscores. The module name is `<prefix>_<group_name_underscored>`.

Example: group name `celery-certs` yields module name `app_celery_certs` with `name = "<service-name-prefix>-celery-certs"`. The resulting Secrets Manager path is `<project>/<env>/<service-name-prefix>-celery-certs/<service-name-prefix>-celery-certs-credentials`.

**For each additional secret, `/sync` ensures:**

1. **Terraform module** exists in `main.tf`, after the component's app secret module:

```hcl
module "<prefix>_<suffix>" {
  source = "./modules/app-secret"

  project     = var.project
  environment = var.environment
  name        = "<service-name-prefix>-<suffix-with-hyphens>"

  common_tags = local.common_tags
}
```

The `app-secret` module creates a Secrets Manager secret with placeholder values (`ignore_changes` on `secret_string`). Terraform manages the secret lifecycle; `/secrets` manages the actual values. Use `/secrets` to set real values after `terraform apply`.

2. **Locals ARN** - add to the component's locals block:
   ```
   <prefix>_<suffix>_secret_arn = module.<prefix>_<suffix>.secret_arn
   ```

3. **Secrets map entries** - for each env var -> key mapping declared in the YAML under this group, add to the component's secrets map:
   ```
   <ENV_VAR> = "${local.<prefix>_<suffix>_secret_arn}:<key>::"
   ```

If no YAML in the component references additional secrets, do not add or remove these blocks.

### ECS Service Module Interface

Every ECS service uses `source = "./modules/ecs-service"`:

```hcl
module "<prefix>_<name>" {
  source = "./modules/ecs-service"

  # Required: platform wiring (same for every service)
  project                 = var.project
  environment             = var.environment
  name                    = "<ecs-service-name>"
  ecs_cluster_arn         = module.ecs.cluster_arn
  task_execution_role_arn = module.iam.ecs_task_execution_role_arn
  task_role_arn           = module.iam.ecs_task_role_arn
  private_subnet_ids      = module.vpc.private_subnet_ids
  security_group_ids      = [module.vpc.ecs_tasks_security_group_id]
  log_group_name          = module.ecs.log_group_name
  aws_region              = var.aws_region

  # Per-service configuration
  container_image = var.<prefix>_<suffix>_image
  container_port  = <port>
  cpu             = var.<prefix>_<suffix>_cpu
  memory          = var.<prefix>_<suffix>_memory
  desired_count   = var.<prefix>_<suffix>_desired_count

  # Optional: override Docker CMD (set only when YAML has entrypoint)
  # command = ["bash", "-c", "..."]

  health_check_path         = "<path>"
  health_check_start_period = 120

  environment_variables = local.<prefix>_env
  secrets               = local.<prefix>_secrets

  # ALB integration (only for publicly-accessible services)
  # enable_alb            = length(module.alb) > 0
  # vpc_id                = module.vpc.vpc_id
  # alb_listener_arn      = length(module.alb) > 0 ? module.alb[0].https_listener_arn : null
  # alb_path_pattern      = "/api*"
  # alb_listener_priority = 100

  common_tags = local.common_tags
}
```

### Variables Pattern

For each ECS service, four variables exist in `variables.tf`:

```hcl
variable "<prefix>_<suffix>_image" {
  type        = string
  default     = "public.ecr.aws/docker/library/busybox:latest"
  description = "Container image URI for <component> <human-name>."
}

variable "<prefix>_<suffix>_cpu" {
  type    = number
  default = <fargate_cpu_units>
}

variable "<prefix>_<suffix>_memory" {
  type    = number
  default = <memory_mib>
}

variable "<prefix>_<suffix>_desired_count" {
  type    = number
  default = <scaling.min_instances>
}
```

The `desired_count` variable is sourced from `scaling.min_instances` in the YAML descriptor. `/sync` sets the default to the YAML value so that Terraform uses it as the baseline task count.

### Unit Conversion and Fargate Constraints

CPU conversion: multiply vCPU by 1024 (e.g. `cpu: 2` becomes `cpu = 2048`).

Memory conversion: multiply GB by 1024, then **snap to the nearest valid Fargate memory value** for the given CPU. Fargate only supports specific CPU/memory combinations. Using an invalid pair causes `RegisterTaskDefinition` to fail.

**Valid Fargate CPU/memory combinations:**

| CPU (units) | Valid memory values (MiB) |
|-|-|
| 256 | 512, 1024, 2048 |
| 512 | 1024, 2048, 3072, 4096 |
| 1024 | 2048, 3072, 4096, 5120, 6144, 7168, 8192 |
| 2048 | 4096, 5120, 6144, 7168, 8192, 9216, 10240, 11264, 12288, 13312, 14336, 15360, 16384 |
| 4096 | 8192 through 30720 (in 1024 increments) |
| 8192 | 16384 through 61440 (in 4096 increments) |
| 16384 | 32768 through 122880 (in 8192 increments) |

**Snapping rule:** After converting `memory_gb` to MiB (multiply by 1024, round to int), find the smallest valid memory value for the given CPU that is >= the computed MiB. If the computed value exceeds the maximum for that CPU, use the maximum.

**Examples:**

| YAML | Raw MiB | CPU units | Snapped memory | Explanation |
|-|-|-|-|-|
| `cpu: 2, memory_gb: 2.3` | 2355 | 2048 | 4096 | 2355 < 4096 (minimum for 2048 CPU) |
| `cpu: 1, memory_gb: 2` | 2048 | 1024 | 2048 | exact match |
| `cpu: 0.5, memory_gb: 3` | 3072 | 512 | 3072 | exact match |
| `cpu: 1, memory_gb: 5.5` | 5632 | 1024 | 6144 | snap up to next valid value |

## YAML Schema

Each file in a component's deploy directory describes one ECS service.

```yaml
service:
  name: <ecs-service-name>
  port: <number>
  dockerfile: <path>
  public_path: <path>         # if present, enables ALB routing
  entrypoint: [...]           # if present, maps to Terraform command

env:
  KEY: value

secrets:
  - rds:                       # maps to RDS credentials secret
      DB_HOST: host
      DB_PORT: port
      DB_PASSWORD: password
  - app:                       # maps to app secret module
      SECRET_KEY: SECRET_KEY
  - <additional-name>:         # maps to additional secret module (auto-managed)
      ENV_VAR: json_key

resources:
  cpu: <vcpu>
  memory_gb: <gb>

scaling:
  min_instances: <n>
  max_instances: <n>

shutdown:
  strategy: rolling            # or drain-queue for workers
  stop_timeout_sec: <seconds>

health_check: <path>

monitoring:
  dashboard: <application-name>
  category: <api|worker|scheduler>
```

### Secret Groups

The `secrets` section uses short group names that `/sync` maps to Terraform resources:

| Group name | Terraform source | Secret ARN local |
|-|-|-|
| `rds` | `module.rds["<key>"].credentials_secret_arn` | `local.<prefix>_rds_secret_arn` |
| `app` | `module.<prefix>_secret.secret_arn` | `local.<prefix>_app_secret_arn` |
| `<custom>` | `module.<prefix>_<custom>.secret_arn` | `local.<prefix>_<custom>_secret_arn` |

Each mapping (`ENV_VAR: json_key`) becomes an entry in the component's `local.<prefix>_secrets` map with the format `${arn}:json_key::`.

### Env Section

The `env` section contains direct environment variables (non-secret values). `/sync` collects the union across all YAML files for a component and rebuilds `local.<prefix>_env`. Known Terraform-expression vars are always preserved.

## Secrets Architecture

The `locals` block for each component has three parts:

1. **ARN definitions** (manually maintained) -- reference Terraform module outputs:
   ```hcl
   <prefix>_rds_secret_arn = module.rds["<key>"].credentials_secret_arn
   <prefix>_app_secret_arn = module.<prefix>_secret.secret_arn
   ```

2. **Env map** (reconciled by `/sync` from YAML `env` sections) -- direct environment variables:
   ```hcl
   <prefix>_env = {
     KEY = "value"
   }
   ```

3. **Secrets map** (reconciled by `/sync` from YAML `secrets` sections) -- ECS secret references:
   ```hcl
   <prefix>_secrets = merge(
     local.<prefix>_rds_secret_arn != "" ? {
       DB_HOST = "${local.<prefix>_rds_secret_arn}:host::"
     } : {},
     {
       SECRET_KEY = "${local.<prefix>_app_secret_arn}:SECRET_KEY::"
     }
   )
   ```

## Secret Value Management

One unified command manages secret **values** in AWS Secrets Manager. It is separate from `/sync`, which manages secret **structure** (module blocks and ARN references in Terraform).

- `/secrets <env>` -- list, create, and edit secrets in AWS Secrets Manager (interactive menu with temp file editing, version labeling, and ECS reload)

### Separation of Concerns

| Concern | Owner | Tool |
|-|-|-|
| Secret structure (module blocks, ARN references in Terraform) | Terraform via `/sync` | `terraform apply` |
| Secret lifecycle (creation with placeholders) | Terraform via `app-secret` module | `terraform apply` |
| Secret values (editing actual credentials) | Developer via `/secrets` | `aws secretsmanager put-secret-value` |

### Value Updates Do Not Require Redeploy

ECS tasks fetch secret values from Secrets Manager at task launch. Updating a value via `/secrets` takes effect on the next ECS task start. No `terraform apply` needed for value-only changes.

### Structural Changes Require `/sync`

Adding or removing a secret **key** (not just changing its value) requires:
1. Update the YAML descriptors in `deploy/<env>/` to reference the new/removed key
2. `/sync` -- reconciles Terraform secret modules, secrets map, and env map from YAML
3. `terraform apply` -- creates any new secret modules (with placeholders)
4. `/secrets` -- set actual values for new or updated keys
5. ECS reload or new deployment to pick up the changes

## Monitoring Module

The `modules/monitoring` module creates a CloudWatch dashboard per application. Each component gets its own monitoring module instance.

### YAML Source

Each YAML service descriptor may have a `monitoring` section:

```yaml
monitoring:
  dashboard: myapp
  category: worker
```

If the `monitoring` section is absent, the service is skipped when building the `ecs_services` map.

### Scope for /sync

When `/sync` adds or removes an ECS service, read the `monitoring` section from each YAML file and update the `ecs_services` map inside the component's monitoring module. The rest of the monitoring module inputs (`rds_instances`, `log_group_name`) are static.

### ecs_services Map Entry Pattern

```hcl
"<human-label>" = {
  service_name = module.<prefix>_<suffix>.service_name
  category     = "<value from YAML monitoring.category>"
}
```

## Existing Module Example

### Template application (webserver with ALB)

```hcl
module "app" {
  source = "./modules/ecs-service"

  project                 = var.project
  environment             = var.environment
  name                    = "${var.project}-app"
  ecs_cluster_arn         = module.ecs.cluster_arn
  task_execution_role_arn = module.iam.ecs_task_execution_role_arn
  task_role_arn           = module.iam.ecs_task_role_arn
  private_subnet_ids      = module.vpc.private_subnet_ids
  security_group_ids      = [module.vpc.ecs_tasks_security_group_id]
  log_group_name          = module.ecs.log_group_name
  aws_region              = var.aws_region

  container_image = var.app_image
  container_port  = 8080
  cpu             = var.app_cpu
  memory          = var.app_memory
  desired_count   = var.app_desired_count

  health_check_path         = "/health"
  health_check_start_period = 120

  environment_variables = local.app_env
  secrets               = local.app_secrets

  enable_alb            = length(module.alb) > 0
  vpc_id                = module.vpc.vpc_id
  alb_listener_arn      = length(module.alb) > 0 ? module.alb[0].https_listener_arn : null
  alb_path_pattern      = "/*"
  alb_listener_priority = 100

  common_tags = local.common_tags
}
```
